## （3）知识使用困难与对策

| 困难描述 | 已解决做法 / 建议 |
|----------|-------------------|
| AI 上下文窗口有限，单文件过大导致无法完整读取设计规范 | **拆分策略**：将 1700+ 行的 `design.md` 拆分为 18 个独立子文档（foundation 7 个 + components 6 个 + pages 4 个 + README 索引），每个文档聚焦单一主题 |
| AI 难以判断"用户查询应该读哪个子文档" | **索引映射表**：在 `README.md` 中建立"关键词 → 目标文件 → 备选文件"的调用映射表，并强制约定"单一文件原则" |
| AI 生成界面时容易遗漏细节（如阴影方向、圆角大小、禁用态透明度） | **Token 化 + 示例代码**：将设计规则转为 CSS 变量和代码片段，AI 可直接复制使用；在 `tokens.md` 中汇总全部变量 |
| AI 使用图标时风格不一致（线型/填充混用）或找不到对应 SVG | **图标索引 + 替代原则**：`icons.md` 列出全部 58 个 SVG 路径；约定找不到时去 Lucide 搜索同类型线性图标，线条粗细 1.5px~2px |
| AI 生成页面时组件组合随意，不符合金融中后台习惯 | **页面模板层**：在 `pages/` 下预定义 4 种典型页面（列表/表单/详情/仪表盘），明确每层结构、引用的组件文档、布局参数 |
| 提示词中业务需求描述过长，AI 容易遗漏控件 | **结构化分层描述**：按"顶部/导航/主内容区/模块/控件"逐层拆解，每个控件标注规格（尺寸、颜色、数据格式） |

## （4）知识梳理/治理做法与困难

| 治理做法 | 困难 | 已解决 / 建议 |
|----------|------|---------------|
| **Figma → Markdown 人工转录**：从 Figma 视觉稿整理并提取（如无组件库）颜色、字体、间距、组件状态，转为结构化 Markdown 表格和代码块 | 工作量大（约 8 大分区、50+ 组件、数百个状态），且 Figma 更新后需重新比对 | 建立"Figma 页面 → Markdown 文件"的一一映射关系（见 `design-specs/README.md` 第 3 节），降低后续维护成本 |
| **设计 Token 统一命名**：颜色按"品牌主色/红/橙/中性色/功能色"分级，间距按 `space-0~16` 编号，避免分散命名 | 初期 Figma 中 Token 命名不统一（如 `点文本-特粗/96pt` 与 `中性色/中性/N3（分隔线）` 混用） | 在 `tokens.md` 中建立统一的 CSS 变量名（如 `--color-primary`、`--space-4`），与 Figma Token 做对照表 |
| **组件状态全枚举**：对每个交互组件（按钮、输入框、菜单等）列出 normal/hover/active/disabled/loading 等全部状态的色值和样式 | 状态组合爆炸（如按钮 7 类型 × 3 尺寸 × 5 状态），文档膨胀 | 采用"类型定义 + 状态表"的模式，共用同一套状态描述，避免重复 |
| **图标 SVG 手工导出与优化**：从 Figma 导出 48 个通用图标 + 13 个状态图标，统一设置 `overflow: visible`、线条宽度 1.5px | SVG 路径在不同尺寸下可能失真；Figma 导出时部分图标未合并路径 | 导出后在代码中统一使用 `shape-rendering: geometricPrecision`，并约定优先使用内联 SVG |
| **页面 Demo 验证**：将规范应用于实际业务页面（证金业务平台仪表盘、私募基金详情页），检验规范可行性 | 规范与实际业务需求存在 gap（如渐变卡片、组合图表在基础规范中未定义） | 在业务 Demo 中补充"渐变样式"、"图表组合卡片"等特殊规则，并反向沉淀到 `components/business.md` |

## （5）知识组织做法与困难

| 组织方式（目录/元数据/分层） | 困难 | 已解决 / 建议 |
|-----------------------------|------|---------------|
| **三级分层目录**：`foundation/`（底层 Token）→ `components/`（中层组件）→ `pages/`（顶层页面模板），外加独立 `icons/` 资源层 | 层级边界有时模糊（如"筛选区"是组件还是页面的一部分？"图表"是基础组件还是业务组件？） | 明确划分原则：foundation 无业务语义；components 可跨页面复用；pages 是完整业务场景。筛选区/图表归入 `components/business.md`，因为跨页面复用 |
| **索引文档（README.md）作为检索入口**：集中管理调用映射表、文件摘要、维护规则 | 索引文档本身也会膨胀，若规则描述过多会消耗 Token | 将约束规则精简为 4 条（单一文件原则、页面优先于组件、Token 溯源优先、图标直达），并附 1 张映射表 + 1 张摘要表 |
| **文件内容摘要**：为每个子文档提供 1 句话摘要，辅助 AI 快速判断调用目标 | 摘要需要随文件内容更新而更新，容易遗漏 | 在 `README.md` 第 5 节集中维护所有文件摘要，每次修改子文档后同步更新 |
| **层级依赖关系图**：用 ASCII 图展示 foundation → components → pages 的引用关系 | 纯文本图难以表达复杂依赖（如 pages 同时引用多个 components） | 简化依赖描述，仅标注"某 page 主要引用哪些 components"，不追求完整依赖图 |
| **版本与日期元数据**：每份文档末尾标注版本号、更新日期、更新人 | 目前仅 `design.md` 和 `tokens.md` 有更新日志，其余文档缺失 | 建议统一在 `README.md` 中维护全局变更日志，避免分散 |

## （6）知识检索做法与困难

| 检索方式 | 困难 | 已解决 / 建议 |
|----------|------|---------------|
| **关键词精确匹配**：在 `README.md` 中建立"用户查询关键词 → 必须调用的文件 → 备选文件"映射 | 用户查询词多样（如"主按钮颜色"vs"主要按钮 hover 色"），无法穷举 | 提取核心语义词（颜色/按钮/表格/列表页等），采用最左匹配策略；在 `tokens.md` 中汇总全部 Token，作为兜底检索目标 |
| **文件摘要快速判断**：AI 先读 README 摘要，再决定调用哪个子文档 | 摘要信息密度有限，某些细节（如"分段控制器 Segmented"）不易从摘要推断 | 在摘要中同时列出文件内的组件/主题清单（如 `business.md` 摘要包含"图表、筛选区、工具栏、底部操作栏"） |
| **图标路径直达**：用户明确询问某个图标时，直接引用 `icons/common/xxx.svg`，不经过 `icons.md` | 图标文件名与设计稿中的中文名称需人工对应（如 `arrow-down.svg` 对应"向下"） | 在 `icons.md` 中同时列出"文件名、图标名称、用途"三列，建立双语索引 |
| **页面模板优先**：若用户询问"某页面怎么设计"，直接调用 `pages/` 下对应文件，不分散查询组件 | 页面文档篇幅较长（如 `list-page.md` 含 12 层结构），可能超出单次读取限制 | 页面文档本身也采用分层标题结构，AI 可定位到具体章节；必要时将页面文档进一步拆分为"结构层"和"细节层" |

| 触发条件 | 责任人 | 更新方式与频率 |
|----------|--------|----------------|

| 季度例行审查 | 设计规范维护人 | 每季度检查一遍 `README.md` 索引与实际文件是否一致，清理失效链接，更新版本号 |

## （10）其他有必要总结的内容

### 10.1 从"单一大文档"到"分层知识库"的演进

- **第一阶段**：`design.md`（约 1780 行）。包含全部规范，但 AI 单次读取无法消化，经常遗漏细节。
- **第二阶段**：`design-spec.md`（约 1110 行）。精简版，但仍为单文件。
- **第三阶段**：`design-specs/` 目录（18 个文件 + 58 个图标）。按 Foundation/Components/Pages/Icons 四层拆分，配合 `README.md` 索引，实现精准单文件调用。

### 10.2 规范落地的验证方式

- **静态验证**：AI 根据规范生成 HTML Demo ，人工检查颜色、间距、组件状态是否符合 Figma 中的设计规范、组件库。
- **动态验证**：将规范应用于真实前端项目（React/Vue），检查 Token 在代码中的可执行性（如 CSS 变量是否覆盖全场景）。

### 10.3 与业务需求的耦合程度

- 基础层（foundation）完全与业务无关，可复用到其他金融系统。
- 组件层（components）90% 通用，仅"业务组件"（图表、筛选区、工具栏）带金融中后台语义。
- 页面层（pages）的页面结构（列表/表单/详情/仪表盘）可跨业务复用。

### 10.4 当前知识库规模

| 层级 | 文件数 | 图标数 | 覆盖组件/页面数 |
|------|--------|--------|-----------------|
| foundation | 7 个 md | — | 颜色/字体/间距/阴影/圆角/图标/tokens |
| components | 6 个 md | — | 60+ 组件（含状态变体） |
| pages | 4 个 md | — | 4 种典型页面模板 |
| icons | — | 58 个 SVG | 通用 45 + 状态 13 |
| 合计 | 18 个 md + 1 README | 58 SVG | 约 70+ 设计条目 |

## 成功经验

| 知识侧做法 | 对 AI 辅助质量的提升 | 可复制要点 |
|------------|----------------------|------------|
| **分层拆分 + 索引映射**：将大文档拆分为 18 个子文档，通过 README 关键词映射实现精准调用 | AI 不再"囫囵吞枣"，每次只读最相关的 1 个文件，输出准确率大幅提升；Token 消耗降低约 60% | 任何领域知识库都可采用"3~4 层分层 + 中央索引"模式；索引中必须包含"关键词→文件"的显式映射 |
| **设计 Token 化（CSS 变量）**：颜色、间距、阴影、字体全部转为 `--token-name` 形式 | AI 生成代码时直接引用变量，避免硬编码色值；当设计变更时只需改 Token，不用改每处代码 | Token 命名采用"属性+语义"双维度（如 `--color-primary`、`--space-4`），既易读又易检索 |
| **页面模板预定义**：提前定义列表/表单/详情/仪表盘 4 种页面的完整结构层次 | AI 接到业务需求时，先匹配页面模板，再填充组件，避免"从零乱搭"；输出结构符合金融中后台用户习惯 | 页面模板应包含"第 N 层结构 + 该层引用的组件文档 + 布局参数"，形成可执行的组装说明书 |
| **结构化需求提示词模板**：按"顶部→导航→主内容→模块→控件"五层拆解需求，每个控件绑定规格 | AI 不会遗漏控件（如"时间选择器"、"分段控制器"）；生成的界面与需求一一对应 | 需求描述采用"层级缩进 + 粗体标注关键规格"的格式，模拟设计稿标注风格 |
| **图标库独立资源层**：58 个 SVG 按 common/status 分类，附索引和替代原则 | AI 能准确找到图标路径，且风格一致；不会出现"用 emoji 代替图标"或"风格混用"的问题 | 图标资源与文档分离，文档中只存索引；替代原则明确到线条粗细和尺寸 |

## 不成功经验

| 已做做法 | 仍无效的主要原因 | 建议 |
|----------|------------------|------|
| 初期尝试用单一大文档（`design.md`）承载全部规范 | AI 上下文窗口有限，读取大文档后常遗漏后半部分；组件状态、页面细节被忽略 | 必须拆分。单文档超过 500 行就应考虑分层 |
| 在提示词中直接粘贴大量设计规范文本 | 提示词中混杂需求描述和规范文本，导致 AI 混淆"要做什么"和"怎么做" | 将规范封装为 Skill（知识库），与需求分离；AI 先读 Skill 建立上下文，再读需求执行任务 |
| 未对 Figma 图层命名做规范化治理，直接按 Figma 原始命名转录 | Token 命名混乱（如 `点文本-特粗/96pt` 与 `中性色/中性/N3（分隔线）` 风格不一致），AI 难以建立语义关联 | 在转录阶段进行二次命名，建立统一的 CSS 变量命名体系，与 Figma 原始命名做对照表即可 |
| 图标导出后未统一优化，部分 SVG 路径存在冗余节点 | AI 在代码中内联 SVG 时，冗余路径导致文件膨胀；部分图标在 16px 下显示模糊 | 导出后统一规范：`overflow: visible`、线条 1.5px、`shape-rendering: geometricPrecision`；优先使用内联而非外部引用 |
